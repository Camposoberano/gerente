Gerente Router V2 Fase 6 concluida

Passo a Passo Operacional do Gerente

Este documento define como o gerente vai evoluir a partir de agora: primeiro planeja, depois roteia, depois executa com fiscal, e por fim registra aprendizado para qualquer projeto, seja EvoHub, Portainer, VPS, Supabase, GitHub ou site.

Contrato Do Comando

O comando principal segue o contrato da skill gerente.

/gerente <tarefa>
/gerente produto <tarefa>
/gerente negocio <tarefa>
/gerente ajuda

Se faltar tarefa, a resposta padrao deve ser:

Use: /gerente <tarefa> | /gerente produto <tarefa> | /gerente negocio <tarefa> | /gerente ajuda

Objetivo

Fazer o gerente virar a central unica de operacao de agentes. Ele nao pertence ao EvoHub; ele atende qualquer projeto e escolhe agente, modelo, fiscal, custo e ambiente conforme a tarefa.

Regra De Seguranca

Acoes de alto risco nao rodam direto. Deploy, escrita em banco, segredo, DNS, exclusao e migracao real precisam de aprovacao explicita e fiscal obrigatorio.

Fluxo A Partir De Agora

1

Receber o comando

O usuario chama /gerente por Codex, Claude Code, dashboard ou futuramente WhatsApp/audio.

  • Preservar a intencao original.
  • Identificar se foi automatico, produto ou negocio.
  • Bloquear comando vazio com a mensagem padrao.
2

Gerar plano no Router V2

Antes de executar, o Router V2 cria um execution_plan.

  • Classifica area: codigo, visual, infra, data, automacao, seguranca, conteudo, review ou ops.
  • Classifica risco: baixo, medio ou alto.
  • Classifica complexidade: baixa, media ou alta.
  • Define gerente, executor, fiscal e fallback.
3

Aplicar politica de custo-beneficio

O gerente nao deve escolher o modelo mais barato nem o mais caro por padrao. Ele escolhe o melhor custo-beneficio para a funcao.

  • Resumo simples: modelo barato/rapido.
  • Frontend e conteudo: modelo eficiente com boa qualidade.
  • Codigo, infra e banco: modelo forte quando o risco subir.
  • Premium so entra com justificativa, falha ou fiscal reprovando.
4

Mostrar plano antes de executar

Na Fase 2, o /gerente mostra um resumo do plano antes de chamar o gerente e o executor.

  • Area detectada.
  • Risco e travas.
  • Executor e fiscal.
  • Ferramentas permitidas e bloqueadas.
5

Executar em lane controlada

Cada tarefa roda em uma faixa de execucao com escopo definido, para evitar agentes se atrapalhando.

  • Arquivos ou ambientes permitidos.
  • Ferramentas liberadas.
  • Limite de custo.
  • Critério de pronto.
6

Fiscalizar antes de concluir

O fiscal valida a entrega quando o risco for medio ou alto, ou quando o executor travar.

  • Verifica escopo.
  • Procura erro tecnico.
  • Confere se houve teste ou verificacao real.
  • Pede fallback quando necessario.
7

Registrar resultado e aprendizado

Depois da execucao, o gerente grava o que funcionou, custo, falhas, agente usado, modelo usado e qualidade da entrega.

  • Atualiza ranking interno por funcao.
  • Registra falhas recorrentes.
  • Melhora roteamento futuro.
  • Alimenta dashboard e notificacoes.

O Que E A Fase 5

A Fase 5 cria runners por ambiente. O Router V2 decide qual ambiente a tarefa pertence, mas o runner e quem controla como agir naquele ambiente sem misturar responsabilidades.

Runner Funcao Limite inicial
repo_local Ler arquivos, aplicar alteracoes e rodar testes no projeto local. Somente dentro da pasta do projeto autorizado.
portainer Consultar stacks, containers, logs e status de servicos. Diagnostico primeiro; deploy exige aprovacao.
vps Executar diagnosticos remotos seguros em VPS. Comandos destrutivos bloqueados por padrao.
supabase Ler schema, validar queries e preparar migrations. Escrita em banco exige aprovacao explicita.
github Consultar issues, PRs, checks, CI e historico de mudancas. Merge, push e release exigem gate de revisao.
coolify Consultar deploy, logs, health e estado do servico. Redeploy e rollback exigem aprovacao.

Fase 6 Implementada

A Fase 6 criou um dashboard local e uma camada de notificacoes. O canal WhatsApp fica pronto para uso com WhatsApp Go/Uazapi via variaveis de ambiente, sem guardar segredo no codigo. 

npm run dashboard
npm run test:notify
Canal Como funciona Configuração
local Registra toda notificacao em .agents/notifications.json. Sempre ativo.
webhook Envia JSON para um endpoint HTTP externo. NOTIFY_WEBHOOK_URL e token opcional.
whatsapp_go Envia texto via /send/text com header token. WHATSAPP_GO_URL, WHATSAPP_GO_INSTANCE_TOKEN, WHATSAPP_NOTIFY_TO.
